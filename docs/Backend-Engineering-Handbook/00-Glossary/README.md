# Glossary

The glossary serves as a quick-reference dictionary for the Backend Engineering Handbook.

Its purpose is to provide concise definitions for technical terms encountered throughout the handbook without requiring readers to search through individual chapters.

Unlike the handbook chapters, glossary entries are intentionally brief. Each entry answers the question:

> "What does this term mean?"

For a deeper explanation, examples, and mental models, refer to the corresponding handbook chapter.

## How to Use This Glossary

- Use the glossary when you encounter an unfamiliar term.
- Use it as a quick refresher before interviews.
- Add new terms whenever a new concept is introduced.
- Keep definitions short, clear, and technology-agnostic whenever possible.

## What Belongs Here

Examples include:

- API
- DbContext
- DbSet
- Dependency Injection
- DTO
- Entity
- Interface
- LINQ
- Middleware
- Migration
- ORM
- Pagination
- Repository
- Service Layer
- Task

The glossary should define terminology, not teach complete concepts. If an explanation grows beyond a few paragraphs, it likely belongs in its own handbook chapter instead.

## Relationship to the Handbook

The handbook explains **why** concepts exist and **how** they work.

The glossary simply explains **what** they are.

For example:

- **Glossary:** "Repository — A class responsible for retrieving and storing data."
- **Handbook:** Explains the Repository Pattern, why it exists, where it fits in the architecture, examples, common mistakes, interview answers, and mental models.

Together, the handbook and glossary provide both quick reference and in-depth learning.

## Goal

The goal is to build a personal technical dictionary that grows alongside the handbook, making it easy to review terminology, reinforce understanding, and quickly revisit concepts learned throughout the backend engineering journey.
